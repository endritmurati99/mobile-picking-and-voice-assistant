## Bugs gefunden & behoben

### Bug 1: button_validate zu früh aufgerufen
- **Problem:** Nach dem Bestätigen der ersten Pick-Zeile wurde das Picking sofort auf `done` gesetzt — alle weiteren Zeilen verschwanden.
- **Ursache:** `confirm_pick_line()` in `picking_service.py` prüfte `all_done` mit einem `or`-Fallback: `move.get("quantity_done", 0) >= move.get("quantity", 0)`. Da die geladenen Felder nur `["picked"]` enthielten, waren `quantity_done` und `quantity` immer `0` → `0 >= 0 = True` → alle Moves galten als fertig.
- **Fix:** Fallback entfernt. Nur noch `all(move.get("picked") for move in moves)`.
- **Status:** ✅ Gefixt, Tests aktualisiert

### Bug 2: Voice-Modus blieb bei Navigation aktiv
- **Problem:** M-Taste oder Mic-Button aktivierten den Voice-Modus in der Detail-Ansicht. Beim Navigieren (zurück zur Liste, Quality-Alert öffnen) blieb das Mikrofon aktiv, obwohl der Button versteckt war.
- **Ursache:** `document.addEventListener('keydown', ...)` ist global und wird nie entfernt. `updateToolbar()` versteckte zwar den Button, stoppte aber den Voice-Modus nicht.
- **Fix:** `stopVoiceMode()` in `voice.js` exportiert. In `updateToolbar(view)` wird `stopVoiceMode()` aufgerufen wenn `view !== 'detail'`.
- **Status:** ✅ Gefixt

## Neue Features

### Voice-First UI
- **Proaktive Begrüßung:** App spricht beim Öffnen automatisch: "X Aufträge offen. Y davon dringend."
- **Prioritäts-Filter:** Buttons `[Alle]` und `[⚡ Dringend]` in der Picking-Liste. Filtert Pickings nach Odoo-Priority `'2'` oder `'3'`. Count-Badge rechts: "2 von 7".
- **Voice-Intents für Filter:**
  - "dringend" / "hohe Priorität" / "kritisch" → `filter_high` → Filter aktiv
  - "alle" / "zurücksetzen" → `filter_normal` → Filter weg
  - "wie viele" / "status" / "übersicht" → `status` → spricht Anzahl offen + dringend

### Neue Voice-Kommandos
| Befehl | Intent | Aktion |
|--------|--------|--------|
| "foto" / "kamera" | `photo` | Kamera-Scanner öffnen |
| "pause" / "stopp" | `pause` | Voice-Modus deaktivieren |
| "fertig" | `done` | Wenn alle bestätigt → zur Liste; sonst "Noch X ausstehend" |
| "dringend" | `filter_high` | Nur dringende Pickings zeigen |
| "alle" / "zurücksetzen" | `filter_normal` | Filter aufheben |

| "noch da" / "lagerbestand" | `stock_query` | Lagerbestand für aktuelles Produkt abfragen |

### Audio-Feedback (`feedback.js`)
- Rein Web Audio API — kein externes Asset, kein Server
- `feedbackSuccess()` → heller Doppel-Piep (880Hz → 1100Hz) + kurze Vibration `[50,30,50]`
- `feedbackError()` → tiefer Brummton (180Hz, Square-Wave) + lange Vibration `[200,100,200]`
- `feedbackAlert()` → markante 3-Ton-Folge (für spätere Eilauftrag-Benachrichtigung)
- Ausgelöst bei Scan: Erfolg → `feedbackSuccess`, falsch/Fehler → `feedbackError`

### Bestandsabfrage

- **Funktion:** Fragt `stock.quant` in Odoo ab, gibt `quantity_available` + `quantity_total` zurück
- **Voice:** "Noch da?" → Backend → "Laut System sind 5 Stück verfügbar." oder "Kein Bestand. Soll ich einen Qualitätsalarm auslösen?"

## Entscheidungen

> [!info] Voice-First statt "App mit Voice-Feature"
> Das UI ist ab jetzt primär visuelle Bestätigung — der Assistent führt aktiv durch den Workflow.
> Proaktive Begrüßung, Filter per Sprache, Audio-Feedback und Bestandsabfrage sind erste Schritte in diese Richtung.

> [!info] Audio ohne externe Assets
> Web Audio API generiert Töne direkt im Browser via Oszillatoren. Keine WAV-Dateien, kein Server-Request, funktioniert auch offline.

> [!info] button_validate-Fallback entfernt
> Der alte Fallback `quantity_done >= quantity` war ein Workaround aus der gestrigen Session. Nach dem Umstieg auf `picked`-Flag ist er nicht mehr nötig und war aktiv schädlich.

> [!info] Obsidian als Source of Truth, Ruflo als Cache
> Obsidian bleibt die lesbare Wissensbasis (Human-Readable Knowledge), während der über `ruflo init` gestartete Daemon einen maschinenlesbaren **Vektor-Index** darüberlegt. Dadurch wird Context-Sprawl vermieden und Token-Verbrauch drastisch reduziert, da die KI gezielt auf Kontextfragmente aus der Architektur zugreifen kann.

> [!info] n8n Webhook Workflow für Quality Alerts
> Um die PWA performant zu halten und die Orchestration asynchron abzubilden, lauscht n8n auf Odoo-Aktionen (`quality.alert` created) und steuert von dort E-Mails sowie den Obsidian-Log an. Evaluierung ist dokumentiert unter [[Webhook vs Polling Evaluierung]].

### Fixes dieser Session

#### Service Worker interceptet API-Calls (sw.js)
- **Problem:** `respondWith(fetch(...).catch(() => caches.match(...)))` liefert `null` für `/api/*` (nicht im Cache) → iOS: "Fetch event.respondWith received an error; returned respond is null"
- **Fix:** `if (e.request.url.includes('/api/')) return;` — API-Calls direkt ans Netzwerk, kein SW-Intercept
- **Cache-Version:** `picking-v1` → `picking-v2` mit Cleanup alter Caches im `activate`-Event

#### iOS TTS: Gesture-Gate + cancel()-Race-Condition
- **Problem:** `speechSynthesis.speak()` blockiert ohne User-Gesture auf iOS Safari
- **Fix:** Dummy-Utterance bei erstem `touchstart`/`pointerdown` (Volume 0.01), queued Speech wird nach 200ms replayed. 80ms Delay nach `cancel()` vor neuem `speak()`.

#### iOS Mic: getUserMedia schlägt still fehl
- **Problem:** Button blinkt nicht rot, sofort "Sprachmodus beendet". Keine Fehlermeldung.
- **Fix:** `toggleVoiceMode` akzeptiert dritten `onError`-Callback. Bei `NotAllowedError` → Toast mit Verweis auf iOS-Einstellungen → Datenschutz → Mikrofon → Safari.

#### iOS TTS Stimmqualität
- **Problem:** iOS wählt "Eloquence"-Stimme (roboterhaft) statt Enhanced-Stimme
- **Fix:** `_loadBestDeVoice()` priorisiert: Markus/Anna Enhanced → beliebig Enhanced → Markus/Anna Compact → kein Eloquence
- **Offener Punkt:** Nutzer muss Enhanced-Stimmen manuell herunterladen (Einstellungen → Bedienungshilfen → Gesprochene Inhalte → Stimmen → Deutsch)

#### Android Voice Debug-Feedback
- **Problem:** Sprachbefehle "nicht erkannt" ohne Hinweis warum
- **Fix:** Toast zeigt jetzt `"text → intent"` oder `"text → nicht erkannt"` (warning)

#### Priority-Filter Backend
- **Problem:** `get_open_pickings()` lieferte `priority`-Feld nicht mit, Frontend prüfte falsch gegen `'2'`/`'3'`
- **Fix:** Backend gibt `priority` zurück, Frontend filtert korrekt auf `'1'` (Odoo: 0=Normal, 1=Dringend)

#### Seed-Daten mit Diversität
- 8 Pickings erzeugt: 4 Normal (heute/morgen/übermorgen), 4 Dringend (heute/gestern/3 Tage überfällig)
- 1 Picking mit 5 Positionen, 1 mit 4 Positionen
